Suppose you want to travel from **Hyderabad to Bangalore**.

Before you start driving, you first decide:

* Which roads to take

You **plan first**, then **travel**.

# Spark Example

Suppose you write:

```python
rdd = sc.textFile("data.txt")

rdd1 = rdd.map(lambda x: x.upper())

rdd2 = rdd.filter(lambda x: "A" in x)

rdd3 = rdd.reduceByKey(lambda a,b: a+b)

rdd3.collect()
```

---

## Step 1: Spark Does NOT Execute Immediately

When Spark sees:

```python
map()

filter()

reduceByKey()
```

it **does nothing**.

Instead, it simply notes:

> "Okay, first do `map`, then `filter`, then `reduceByKey`."

This list of operations is called **Lineage** (the logical history).

---

# Step 2: `collect()` is Called

Now you say:

```python
rdd3.collect()
```

Spark says:

> "Now I actually need the result. Let me figure out the best way to execute these operations."

This is where the **Execution Plan** is created.

---

# What is an Execution Plan?

An **Execution Plan** is simply:

> **Spark's strategy for executing your code efficiently.**

Instead of blindly running each line one by one, Spark creates a smart plan.

Think of it like:

```text
Your Code
    ↓
Spark thinks...
    ↓
"How can I execute this faster?"
```

---

# What is a DAG?

A **DAG (Directed Acyclic Graph)** is the **visual representation of Spark's execution plan**.

It shows:

* Which operations happen first
* Which operations can be combined
* Where data needs to be shuffled
* Where new stages begin

So:

```text
Execution Plan = Spark's strategy

DAG = Picture of that strategy
```

---

# Example

Your code:

```python
rdd = sc.textFile("data.txt")

rdd1 = rdd.flatMap(...)

rdd2 = rdd1.map(...)

rdd3 = rdd2.reduceByKey(...)

rdd3.collect()
```

Spark internally draws something like:

```text
Read File
     |
 flatMap
     |
    map
     |
-----------------
   Shuffle
-----------------
     |
reduceByKey
     |
 collect
```

This picture is the **DAG**.

---

# Why Does Spark Create a DAG?

Imagine Spark executes every operation separately.

```text
Read File

↓

Store Result

↓

Map

↓

Store Result

↓

Filter

↓

Store Result
```

This is slow.

Instead, Spark says:

> "Why don't I combine `map` and `filter` into one step?"

So it executes:

```text
Read File

↓

Map + Filter Together

↓

Shuffle

↓

Reduce
```

Much faster!

That's why Spark creates an execution plan (DAG).

# DAG vs Lineage

This is where many students get confused.

### Lineage

Lineage answers:

> **"How was this RDD created?"**

Example:

```text
RDD0

↓

map

↓

RDD1

↓

filter

↓

RDD2
```

It is mainly used for **fault tolerance**. If an RDD is lost, Spark can recreate it using this history.

---

### DAG

DAG answers:

> **"How should I execute these operations efficiently?"**

Example:

```text
Stage 0

Read File

↓

Map

↓

Filter

----------------

Shuffle

----------------

Stage 1

↓

ReduceByKey
```

It is mainly used for **execution optimization**.

---

# Where Does the Execution Plan Come From?

Spark follows these steps:

```text
You write code

↓

Spark records Lineage

↓

Action (collect, count, show)

↓

Spark creates Execution Plan (DAG)

↓

Creates Stages

↓

Creates Tasks

↓

Runs on Executors
```

# Easy Analogy: Cooking 🍳

Suppose you want to make fried rice.

### Recipe (Lineage)

```text
Boil rice

↓

Cut vegetables

↓

Cook vegetables

↓

Mix rice

↓

Serve
```

This is just the list of steps.

---

### Cooking Plan (Execution Plan)

You think:

* While rice is boiling, cut vegetables.
* Then cook vegetables.
* Finally mix everything.

This optimized workflow is the **Execution Plan**.

---

### Kitchen Workflow Diagram (DAG)

```text
Boil Rice
     \
      \
       Mix
      /
Cook Vegetables
```

That's the DAG.

# Simple Difference Table

| Lineage                              | DAG (Execution Plan)                                 |
| ------------------------------------ | ---------------------------------------------------- |
| Records transformation history       | Decides how to execute the transformations           |
| Used for fault tolerance             | Used for performance optimization                    |
| Logical plan                         | Physical execution plan                              |
| Created as you write transformations | Created when an action is called                     |
| Shows parent-child RDD relationships | Shows stages, shuffle boundaries, and execution flow |

---

# Complete Spark Flow

```text
You write transformations
        │
        ▼
   Lineage Created
(Logical history of RDDs)
        │
        ▼
 Action (collect, count...)
        │
        ▼
 DAG Scheduler Creates
 Execution Plan (DAG)
        │
        ▼
 Breaks Job into Stages
        │
        ▼
 Creates Tasks
        │
        ▼
 Executors Execute Tasks
        │
        ▼
     Final Result
```

# One-Line Definitions (Great for Interviews)

* **Lineage:** *A logical record of all transformations applied to RDDs, used for fault tolerance.*
* **Execution Plan:** *Spark's optimized strategy for executing a job.*
* **DAG:** *A Directed Acyclic Graph that visually represents Spark's execution plan, showing how operations are organized into stages and executed efficiently.*

### The easiest way to remember it is:

* 📝 **Lineage** = "What operations did I write?"
* 🧠 **Execution Plan** = "How should Spark execute them efficiently?"
* 📊 **DAG** = "A diagram of that execution plan."
