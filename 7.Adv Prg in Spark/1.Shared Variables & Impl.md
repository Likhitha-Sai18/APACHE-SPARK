# Advanced Programming in Spark – Shared Variables (Detailed Notes)

In distributed computing, Spark runs tasks on **multiple executors (worker nodes)**. Normally, each task works independently. But sometimes all tasks need to **share some common information**.

Spark provides **Shared Variables** for this purpose.

There are **two types of shared variables**:

1. **Broadcast Variables** (Read-only shared data)
2. **Accumulators** (Write-only counters/sums)

---

# Why Do We Need Shared Variables?

Imagine this Spark program:

```python
countries = ["India", "USA", "UK", "Japan"]

rdd = sc.parallelize(data)

rdd.map(lambda x: process(x, countries))
```

Every task receives its own copy of `countries`.

If the list is very large (say 500 MB),

Spark keeps sending that 500 MB to every executor.

This wastes

* Network bandwidth
* Memory
* Time

Spark solves this using **Broadcast Variables**.

---

# 1. Broadcast Variables

## Definition

A **Broadcast Variable** is a **read-only variable** that is sent **only once** to every executor and then cached there.

Instead of sending the same object repeatedly with every task, Spark broadcasts it once.

Think of it like this:

**Without Broadcast**

```
Driver

Task1 ---> 500 MB
Task2 ---> 500 MB
Task3 ---> 500 MB
Task4 ---> 500 MB
```

Huge network traffic.

---

**With Broadcast**

```
Driver

Broadcast 500 MB once
        |
-------------------------
|          |           |
Executor1 Executor2 Executor3
```

Every executor stores one cached copy.

All tasks running there reuse it.

---

# Real-Life Example

Suppose you're processing **100 million customer transactions**.

Every transaction needs a currency conversion table.

```
USD → INR
EUR → INR
GBP → INR
```

This table never changes during execution.

Instead of sending it with every task,

Broadcast it once.

Every executor can access it.

---

# Creating Broadcast Variable

Syntax

```python
broadcast_var = sc.broadcast(value)
```

Example

```python
countries = ["India", "USA", "Japan"]

bc = sc.broadcast(countries)
```

Now use

```python
bc.value
```

instead of

```python
countries
```

Example

```python
rdd.filter(lambda x: x.country in bc.value)
```

---

# Accessing Broadcast Variable

Always use

```python
broadcast_var.value
```

Example

```python
print(bc.value)
```

Output

```
["India", "USA", "Japan"]
```

---

# Why is it Read-Only?

Executors are not allowed to modify broadcast variables.

Why?

Imagine Executor 1 changes

```
India
```

to

```
Bharat
```

Executor 2 still has

```
India
```

Now different executors have different values.

Spark avoids this inconsistency by making broadcast variables **read-only**.

---

# When Should We Use Broadcast Variables?

Use them when:

* Same data is needed by many tasks.
* Data is relatively large.
* Data does not change.
* Lookup tables.
* Configuration values.
* Dictionaries.
* Machine learning model parameters.
* Country/state mappings.

---

# Advantages

* Less network communication.
* Faster execution.
* Lower memory usage.
* Cached on executors.
* Better performance.

---

# When NOT to Use Broadcast Variables

Don't broadcast

* Frequently changing data.
* Extremely huge datasets that won't fit in executor memory.
* Data used by only one task.

---

# Broadcast Variable Flow

```
Driver

Large Lookup Table
       |
broadcast()
       |
--------------------------
|           |            |
Executor1  Executor2  Executor3
     |           |            |
 Tasks      Tasks       Tasks
```

Every executor has one cached copy.

---

# 2. Accumulators

Broadcast variables allow executors to **read** shared data.

Sometimes we want the opposite.

We want executors to **send information back** to the driver.

This is what **Accumulators** do.

---

## Definition

An **Accumulator** is a variable that **executors can add to**, but **only the driver can read**.

Think of it as a global counter or sum.

---

# Real-Life Example

Suppose you're processing log files.

```
INFO
INFO
WARNING
ERROR
INFO
ERROR
WARNING
```

You want to know

* Total Errors
* Total Warnings

Every executor counts its own partition.

At the end,

Spark combines everything.

---

Example

Partition 1

```
ERROR = 2
WARNING = 3
```

Partition 2

```
ERROR = 1
WARNING = 4
```

Partition 3

```
ERROR = 5
WARNING = 2
```

Driver receives

```
Errors = 8
Warnings = 9
```

---

# Creating an Accumulator

```python
acc = sc.accumulator(0)
```

Here

```
0
```

is the initial value.

---

# Updating Accumulator

```python
acc += x
```

or

```python
acc.add(x)
```

---

# Example

```python
acc = sc.accumulator(0)

rdd = sc.parallelize([1,2,3,4])

def f(x):
    global acc
    acc += x

rdd.foreach(f)

print(acc.value)
```

Output

```
10
```

---

# How Does It Work?

Suppose

```
RDD

1 2 | 3 4
```

Partition 1

```
1+2=3
```

Partition 2

```
3+4=7
```

Driver

```
3+7=10
```

Accumulator value

```
10
```

---

# Important Rule

Executors

* Can **only update** the accumulator.
* Cannot read its value reliably.

Driver

* Can read the final value using:

```python
acc.value
```

---

# Common Uses of Accumulators

* Count errors.
* Count invalid records.
* Count missing values.
* Count processed rows.
* Compute totals.
* Monitor job statistics.

---

# Broadcast Variables vs Accumulators

| Feature    | Broadcast Variable             | Accumulator                        |
| ---------- | ------------------------------ | ---------------------------------- |
| Purpose    | Share read-only data           | Collect information from executors |
| Direction  | Driver → Executors             | Executors → Driver                 |
| Read       | Driver & Executors             | Driver only                        |
| Write      | Driver only (before broadcast) | Executors only (add/update)        |
| Mutable    | No (read-only)                 | Yes (through addition)             |
| Common Use | Lookup tables, configs         | Counters, sums, statistics         |

---

# Easy Way to Remember

* 📦 **Broadcast Variable** = "Send one copy of data to all executors."
* ➕ **Accumulator** = "Collect counts/sums from all executors back to the driver."

---

# Interview Questions

**1. What are shared variables in Spark?**
Shared variables are special variables that allow efficient sharing of data between the driver and executors. Spark provides two types: **Broadcast Variables** and **Accumulators**.

**2. What is a Broadcast Variable?**
A read-only variable that is sent once to each executor and cached for reuse by all tasks on that executor.

**3. Why use Broadcast Variables?**
To avoid sending the same large data repeatedly with every task, reducing network traffic and improving performance.

**4. What is an Accumulator?**
A variable that executors can update (using associative operations like addition), while only the driver can read its final value.

**5. What are Accumulators commonly used for?**
Counting errors, tracking processed records, summing values, and collecting job statistics.

**6. Can executors modify a Broadcast Variable?**
No. Broadcast Variables are **read-only**.

**7. Can executors read the value of an Accumulator?**
No. Executors update it, but only the **driver** should read the final value.

---

### Key Takeaway

**Shared Variables** make Spark applications more efficient:

* **Broadcast Variables** efficiently distribute large, read-only data from the **driver to executors**.
* **Accumulators** efficiently collect aggregated information (like counts or sums) from **executors back to the driver**.




# Shared Variables in Spark – Implementation (Broadcast Variables & Accumulators)

This lecture focuses on **how to implement** the two shared variables in Spark with practical examples.

---

# 1. Broadcast Variables – Implementation

## 🎯 Problem Statement

Suppose you have employee records like this:

| First Name | Last Name | Country | State |
| ---------- | --------- | ------- | ----- |
| James      | Smith     | USA     | CA    |
| Maria      | Jones     | USA     | NY    |
| Robert     | Brown     | IND     | AP    |

Notice that:

* Country names are abbreviated (`USA`, `IND`)
* State names are abbreviated (`CA`, `NY`, `AP`)

We want to convert them into:

| Country       | State          |
| ------------- | -------------- |
| United States | California     |
| United States | New York       |
| India         | Andhra Pradesh |

---

## ❌ Without Broadcast Variable

Every task receives its own copy of the lookup dictionary.

```text
Task 1
Country Dictionary

Task 2
Country Dictionary

Task 3
Country Dictionary
```

The same dictionary is sent repeatedly.

❌ More network traffic

❌ Slower execution

---

## ✅ With Broadcast Variable

Create the dictionary once.

```text
Driver

Country Dictionary
State Dictionary
        │
 broadcast()
        │
──────────────────────────
│          │            │
Executor1 Executor2 Executor3
```

Each executor stores one cached copy.

All tasks reuse it.

---

## Execution Flow

```text
Step 1
Driver creates dictionary

USA → United States
IND → India

CA → California
AP → Andhra Pradesh

        │

Step 2
Broadcast()

        │

Step 3
Executors cache dictionary

        │

Step 4
RDD records use dictionary

USA → United States
CA  → California

        │

Final Output

James Smith United States California
```

---

## Code Flow

```python
country_dict
```

↓

```python
sc.broadcast(country_dict)
```

↓

```python
broadcastCountries.value
```

↓

```python
RDD.map()
```

↓

Updated RDD

---

## Why `.value`?

Broadcast variables store the object internally.

To access it:

```python
broadcastCountries.value
```

Think of

```text
Broadcast Object
        │
      value
        │
Dictionary
```

---

## Memory Trick

📦 **Broadcast = Send once → Read many**

Driver → Executors

```text
Driver
   │
broadcast()
   │
▼ ▼ ▼
Executors
```

Remember:

> **Broadcast sends one copy to every executor.**

# 2. Accumulator – Implementation

## 🎯 Problem Statement

Find the sum of numbers

```text
1 2 3 4 5
```

using Spark.

---

## Step 1

Create accumulator

```python
acc = sc.accumulator(0)
```

Initial value

```text
Accumulator = 0
```

---

## Step 2

Create RDD

```text
1 2 3 4 5
```

Suppose Spark creates

```text
Partition 1

1 2

Partition 2

3 4

Partition 3

5
```

---

## Step 3

Every partition updates accumulator

Partition 1

```text
0 + 1 + 2 = 3
```

Partition 2

```text
0 + 3 + 4 = 7
```

Partition 3

```text
0 + 5 = 5
```

---

## Step 4

Spark combines results

```text
3 + 7 + 5 = 15
```

Driver reads

```python
acc.value
```

Output

```text
15
```

---

# Execution Diagram

```text
Driver

Accumulator = 0

        ▲
        │
Executor 1
+3

        ▲
        │
Executor 2
+7

        ▲
        │
Executor 3
+5

Final Value = 15
```

---

# Why use `global`?

Inside

```python
def countFun(x):
```

Spark executes the function on executors.

To allow executors to update the same accumulator

```python
global acc
```

is used.

Otherwise

Python creates a local variable.

---

# Without Accumulator

Suppose

```python
sum = 0
```

Each executor gets

```text
Executor 1

sum = 0

Executor 2

sum = 0

Executor 3

sum = 0
```

Each executor changes only **its own copy**.

Driver still has

```text
sum = 0
```

or an incorrect result.

That's why normal variables cannot be used for distributed counting.

---

# Empty Partition Example

RDD

```text
Numbers

1 2 3 4 5
```

Partitions

```text
P1 → 1

P2 → 2

P3 → 3

P4 → 4

P5 → 5

P6 → Empty

P7 → Empty

P8 → Empty
```

Suppose

```python
acc.add(1)
```

is executed for every element.

Only partitions containing data execute it.

So

```text
P1 → +1

P2 → +1

P3 → +1

P4 → +1

P5 → +1

P6 → No execution

P7 → No execution

P8 → No execution
```

Final accumulator

```text
5
```

**Key Point:** Empty partitions **do not contribute** to the accumulator because there are no elements to process.

## Broadcast Variable

```text
           Driver
      (Lookup Table)

        broadcast()
             │
   ┌─────────┼─────────┐
   ▼         ▼         ▼
Executor1 Executor2 Executor3
(Read Only)(Read Only)(Read Only)
```

📌 **One copy sent once, reused by all tasks.**

## Accumulator

```text

      │
Executor2  +7
      │
Executor3  +5
      │
      ▼
    Driver
Accumulator = 15
```

📌 **Many executors send updates back to one driver.**

# Quick Revision

| Broadcast Variable               | Accumulator                          |
| -------------------------------- | ------------------------------------ |
| Shares lookup/configuration data | Collects counts, sums, statistics    |
| Driver ➜ Executors               | Executors ➜ Driver                   |
| Read-only                        | Add/update only                      |
| Sent once and cached             | Final value read by driver           |
| Example: Country/State mapping   | Example: Sum of numbers, error count |

## 🧠 10-Second Memory Rule

* 📦 **Broadcast = "One Driver → Many Executors"** (share read-only data)
* ➕ **Accumulator = "Many Executors → One Driver"** (collect results)
