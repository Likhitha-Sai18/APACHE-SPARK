# Coalesce vs Repartition in Spark – Easy Concept Notes

---

# 1. Why Do We Need Coalesce & Repartition?

Spark stores data in **partitions**. Sometimes after transformations:

* You may have **too many partitions** (many small files/tasks).
* You may have **too few partitions** (not enough parallelism).
* Your data may be **skewed** (unevenly distributed).

Spark provides **two methods** to change the number of partitions:

1. **Coalesce**
2. **Repartition**

---

# 2. Coalesce

## What is it?

**Coalesce** is used to **reduce the number of partitions** without performing a full data shuffle.

It simply **merges existing partitions**.

---

## Why is it Needed?

Suppose your data has 8 partitions, but only 4 are needed.

Instead of moving all the data across the cluster, Spark combines nearby partitions.

This makes it **faster and cheaper**.

---

## How Does It Work?

Original:

```text
P1  P2  P3  P4
```

Need only **2 partitions**

```text
P1 + P2  →  New P1

P3 + P4  →  New P2
```

### Visualization

```text
Before

P1   P2   P3   P4

 │    │    │    │
 └────┘    └────┘
      │         │
      ▼         ▼

After

New P1     New P2
```

✅ No full shuffle

✅ Simply merges partitions

---

## Key Characteristics

* ✔ Only **decreases** partitions
* ✔ No full data shuffle
* ✔ Faster than repartition
* ✔ Best for well-balanced datasets

---

# 3. Repartition

## What is it?

**Repartition** is used to **increase or decrease** the number of partitions.

Unlike coalesce, it performs a **full shuffle** to distribute data evenly.

---

## Why is it Needed?

Suppose one partition contains much more data than the others (data skew).

Spark redistributes the records evenly across new partitions.

---

## How Does It Work?

Original:

```text
P1  P2  P3  P4
```

Need **2 partitions**

Spark first collects and redistributes all records.

### Visualization

```text
Before

P1   P2   P3   P4

 \    |   /   /
  \   |  /   /
   \  | /   /
    \ |/   /
   SHUFFLE
      │
      ▼

New P1     New P2
```

✅ Data is evenly balanced

❌ More expensive because of shuffling

---

## Key Characteristics

* ✔ Can increase partitions
* ✔ Can decrease partitions
* ✔ Performs full shuffle
* ✔ Better load balancing

---

# 4. Practical Example

Suppose Spark created **8 partitions**.

### Using Coalesce

```python
df.coalesce(4)
```

Result:

```text
8 Partitions

↓

4 Partitions
```

✔ Success

---

Trying

```python
df.coalesce(16)
```

Result:

```text
Still 8 Partitions
```

❌ Coalesce **cannot increase** partitions.

---

### Using Repartition

Current partitions:

```text
4
```

Command:

```python
df.repartition(10)
```

Result:

```text
10 Partitions
```

✔ Works because repartition supports both increasing and decreasing.

---

# 5. Execution Flow

## Coalesce

```text
Dataset
    │
8 Partitions
    │
Merge Partitions
    │
4 Partitions
```

**No shuffle**

---

## Repartition

```text
Dataset
    │
8 Partitions
    │
Shuffle Data
    │
Redistribute Evenly
    │
10 Partitions
```

---

# 6. When to Use Which?

### Use **Coalesce** when:

* Dataset is already balanced.
* You only want **fewer partitions**.
* You want better performance (no shuffle).

Example:

```text
8 → 6

8 → 4
```

---

### Use **Repartition** when:

* You need **more partitions**.
* Data is skewed.
* You want even distribution.
* You need a drastic reduction (e.g., 100 → 1).

---

# 7. Coalesce vs Repartition

| Feature             | Coalesce                           | Repartition                                  |
| ------------------- | ---------------------------------- | -------------------------------------------- |
| Increase partitions | ❌ No                               | ✅ Yes                                        |
| Decrease partitions | ✅ Yes                              | ✅ Yes                                        |
| Data Shuffle        | ❌ No (or minimal)                  | ✅ Full shuffle                               |
| Speed               | Faster                             | Slower                                       |
| Load Balancing      | Limited                            | Excellent                                    |
| Best Use            | Reduce partitions in balanced data | Handle skew or change partition count freely |

---

# 8. Memory Trick

### 🟢 Coalesce

**Merge without moving data**

```text
P1  P2  P3  P4

↓

P1+P2   P3+P4
```

Think: **Combine = Coalesce**

---

### 🔵 Repartition

**Redistribute all data**

```text
P1  P2  P3  P4

↓

🔄 Shuffle

↓

New Balanced Partitions
```

Think: **Rebuild = Repartition**

---

# 9. Interview Questions

**1. What is Coalesce?**
Coalesce reduces the number of partitions by merging existing ones, usually without a full shuffle.

**2. Can Coalesce increase partitions?**
No.

**3. What is Repartition?**
Repartition changes the number of partitions (increase or decrease) by performing a full data shuffle.

**4. Which is faster?**
Coalesce, because it avoids a full shuffle.

**5. When should you use Repartition?**
When increasing partitions, fixing data skew, or ensuring even data distribution.

---

## ⭐ One-Line Summary

* **Coalesce = Merge partitions (Fast, No shuffle, Only decrease).**
* **Repartition = Redistribute partitions (Shuffle, Balanced, Increase or decrease).**
