# Spark Partitioning – Complete Concept (Easy to Understand)

---

# 1. What is Partitioning?

## Definition

**Partitioning** is the process of dividing a large dataset into smaller pieces called **partitions** so Spark can process them in parallel across multiple executors.

Think of partitions as **small chunks of data**.

---

# 2. Why is Partitioning Needed?

Imagine you have **1 TB** of data.

If one computer processes all of it:

❌ Very slow

❌ High memory usage

❌ Cannot utilize multiple machines

Instead, Spark splits it into partitions.

Example:

```text
1 TB Data

        │
        ▼

+-----+-----+-----+-----+
| P1  | P2  | P3  | P4  |
+-----+-----+-----+-----+

        │
        ▼

Executor1 Executor2 Executor3 Executor4
```

Now all executors work **simultaneously**.

✅ Faster execution

✅ Parallel processing

✅ Lower memory usage per machine

---

# 3. How Does Spark Partition Data?

### Step 1

Driver creates an RDD/DataFrame.

```text
Large Dataset
```

↓

### Step 2

Spark divides it into partitions.

```text
Dataset

┌────┬────┬────┬────┐
│ P1 │ P2 │ P3 │ P4 │
└────┴────┴────┴────┘
```

↓

### Step 3

Partitions are assigned to executors.

```text
P1 → Executor 1

P2 → Executor 2

P3 → Executor 3

P4 → Executor 4
```

↓

### Step 4

Executors process partitions independently.

↓

### Step 5

Driver collects the final result.

# 4. Important Properties of Partitions

### Property 1

A partition **never spans multiple nodes**.

```text
Correct

Node 1
P1

Node 2
P2
```

❌ Never

```text
Node 1
Half of P1

Node 2
Half of P1
```

---

### Property 2

One node can contain multiple partitions.

```text
Executor

P1

P2

P3
```

---

### Property 3

Only **one partition is processed by one core at a time**.

Example:

8 cores

```text
Core1 → P1

Core2 → P2

Core3 → P3

...

Core8 → P8
```

---

### Property 4

Default number of partitions

Spark usually creates partitions equal to the **total number of available cores**.

Example

```text
Machine

8 CPU Cores

↓

Default Partitions = 8
```

# Types of Partitioning

Spark mainly uses:

1. Hash Partitioning
2. Range Partitioning

---

# Hash Partitioning

## What is it?

Hash Partitioning distributes records based on the **hash value of the key**.

Spark computes

```text
Partition = Hash(Key) % Number of Partitions
```

---

## Why is it Needed?

To ensure records with the **same key** always go to the **same partition**.

This is useful for operations like:

* join
* groupByKey
* reduceByKey

---

## Execution Flow

```text
(Key, Value)

↓

Hash(Key)

↓

Hash % Partitions

↓

Partition Number

↓

Executor
```

---

## Example

Suppose we have:

```text
Keys

10

25

18

30

42
```

Number of partitions = 4

Formula

```text
Partition = Key % 4
```

Calculation

| Key | Partition |
| --- | --------- |
| 10  | 2         |
| 25  | 1         |
| 18  | 2         |
| 30  | 2         |
| 42  | 2         |

Result

```text
Partition 0

Empty

Partition 1

25

Partition 2

10

18

30

42
```

Notice

Most data went into Partition 2.

# Problem 1 — Data Skew

## What is Data Skew?

Some partitions receive much more data than others.

Example

```text
Partition 1

100 Records

Partition 2

20 Records

Partition 3

10 Records

Partition 4

5 Records
```

---

### Why is it a Problem?

Spark waits for **all partitions** to finish.

```text
P1 ████████████

P2 ██

P3 █

P4 █
```

The stage finishes only after **P1** completes.

So,

Large partition = Slow job.

---

# Problem 2 — Scheduling Problem

Suppose

3 partitions

2 executors

```text
Executor1

P1

↓

P3

Executor2

P2
```

Executor 2 finishes early.

Executor 1 still works.

```text
Executor1 ██████████

Executor2 ███ Idle...
```

One executor stays idle while the other is busy.

This reduces cluster utilization.

---

# Range Partitioning

## What is it?

Instead of using hash values,

Spark divides data into **continuous ranges**.

Example

Age

```text
0–20

21–40

41–60

61–80
```

Each range becomes one partition.

---

## Why is it Needed?

Similar values stay together.

Useful for:

* Sorting
* Range queries
* Time-series data
* Numeric data

---

## Example

Marks

```text
10

15

25

30

45

50

75

90
```

Range Partition

```text
Partition 1

0–25

10

15

25

Partition 2

26–50

30

45

50

Partition 3

51–100

75

90
```

Similar values remain together.

# Hash vs Range Partitioning

| Feature                 | Hash Partitioning             | Range Partitioning                    |
| ----------------------- | ----------------------------- | ------------------------------------- |
| Based On                | Hash(Key)                     | Value Ranges                          |
| Formula                 | Hash % Partitions             | Range Boundaries                      |
| Similar Values Together | ❌ No                          | ✅ Yes                                 |
| Load Balancing          | Usually Good                  | Usually Better if ranges are balanced |
| Best For                | Join, groupByKey, reduceByKey | Sorting, filtering, range queries     |

# Interview Questions

**1. What is a partition in Spark?**
A partition is a smaller chunk of data that Spark processes in parallel across executors.

**2. Why does Spark partition data?**
To enable parallel processing, reduce memory usage per executor, and improve performance.

**3. Can a partition span multiple nodes?**
No. A partition always resides entirely on a single node.

**4. What is Hash Partitioning?**
It distributes records based on the hash of the key so that identical keys go to the same partition.

**5. What is Range Partitioning?**
It groups records into partitions based on consecutive ranges of values.

**6. What is Data Skew?**
A situation where one or a few partitions contain much more data than others, causing slower execution.

**7. What is the Scheduling Problem?**
When partitions are unevenly distributed among executors, some executors become idle while others continue working, reducing resource utilization.
